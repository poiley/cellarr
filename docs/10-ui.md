# 10 — UI: built exclusively from Sacred (SRCL)

The cellarr web UI is composed **exclusively** from components in
[`internet-development/www-sacred`](https://github.com/internet-development/www-sacred) — the
`srcl` library, live at [sacred.computer](https://www.sacred.computer/). This is a hard rule.

## The rule

- **Every screen is assembled only from existing SRCL components.** No bespoke UI primitives, no
  second component library, no hand-rolled design system, no ad-hoc CSS beyond layout glue that
  uses SRCL's own theming tokens. (The one allowed "non-component" addition is the **theme
  controller** that selects SRCL's `theme-light`/`theme-dark` body class from the system preference
  and a persisted override — see Theming below. It introduces no UI primitives.)
- **Light + dark + system default are required.** Every screen must work in both SRCL themes and
  follow the OS preference by default with a manual override (see Theming).
- If a screen seems to need something SRCL doesn't have, **compose it from existing SRCL
  components** first. Only if that is genuinely impossible do you propose adding a component
  *upstream to SRCL's conventions* — and that is a flagged decision, not a unilateral one.
- SRCL is **MIT licensed** (verified) — compatible with our use. It is **Next.js 16 / React 19**.

## How to get the components

- The library is already cloned at `reference/www-sacred/` (study only; see below for how we
  consume it).
- The canonical catalog is `reference/www-sacred/components/AGENTS.md` — **one entry per component**
  with its props, theming tokens, and CLI-primitive equivalent. Read it before picking a component.
- Raw source for any component is also fetchable without cloning:
  `https://sacred.computer/llm/components/<Name>.tsx.txt`. Full docs index:
  `https://sacred.computer/llms.txt` and `llms-full.txt`.

## How cellarr consumes SRCL (consumption policy)

`web/` is a Next.js app. SRCL is consumed as the component layer. Decide **one** of these at
scaffold time and record it in [`specs/web-ui.md`](specs/web-ui.md):

- **Preferred:** add `srcl` as a dependency (it's published to npm as `srcl`) and import components,
  OR vendor the `components/`, `common/`, `modules/`, and global CSS into `web/` following SRCL's
  structure. Vendoring gives version stability; npm gives easy updates. Either way the rule is the
  same: **use SRCL components, do not invent new ones.**
- Bring over SRCL's `global.css` / `global-fonts.css` and its `colors.json`-derived theming tokens
  so the terminal aesthetic and monospace spacing are exact, not approximated.

> SRCL also ships a zero-dependency **CLI framework (Simulacrum)** that mirrors the React
> components in the terminal. This is a natural fit for a future `cellarr` TUI, but the TUI is **out
> of scope for v1** — noted here only so we don't reinvent it later.

## Theming: light, dark, and system default (required)

cellarr's UI **must support both light and dark themes and default to the operating-system
preference**, with a manual override. This is a first-class requirement, not a nice-to-have.

How it maps onto SRCL (use SRCL's own mechanism — do not invent a parallel theming system):

- SRCL drives all color through `--theme-*` tokens selected by a body class: **`body.theme-light`**
  and **`body.theme-dark`** (with matching `tint-*` ramps for both modes), all derived from one
  palette (`colors.json` → `--ansi-*` → `--theme-*`). Components already consume these tokens, so
  both modes "just work" once the body class is set.
- **System default:** on load, read `window.matchMedia('(prefers-color-scheme: dark)')` and apply
  `theme-dark`/`theme-light` accordingly; subscribe to its `change` event so the UI follows the OS
  live (e.g. an OS day/night switch) while the user is on "System".
- **Manual override + persistence:** a three-way setting — **System / Light / Dark** — persisted
  (localStorage); "System" re-enables the `matchMedia` follow behavior, Light/Dark pin the class.
  Build the toggle from existing SRCL controls (e.g. `RadioButtonGroup` or `DropdownMenu`); do **not**
  add a new component.
- **No flash of wrong theme:** set the initial body class before first paint (an inline pre-hydration
  script that reads the stored choice, falling back to `prefers-color-scheme`), and set the CSS
  `color-scheme` property so native form controls/scrollbars match.
- This is **app glue around SRCL's classes/tokens**, so it does not violate the SRCL-only rule. The
  only "new" code is the theme controller (class selection + persistence + the toggle wiring), which
  contains no UI primitives of its own.

Every screen must be legible and correct in **both** modes — verified by the theme tests below.

## The component catalog (what we have to work with)

66 components. The ones cellarr will lean on most, grouped by role:

- **Layout & shell:** `SidebarLayout`, `Navigation`, `Window`, `Card`, `CardDouble`, `Row`,
  `RowSpaceBetween`, `Grid`, `Block`, `Divider`, `ContentFluid`, `Indent`, `DefaultLayout`/
  `DefaultActionBar` (under `components/page/`).
- **Tables & lists (the heart of a media manager):** `DataTable`, `SimpleTable`, `Table` +
  `TableRow`/`TableColumn`, `TreeView` (perfect for Series→Season→Episode and Artist→Album→Track),
  `ListItem`, `ActionListItem`, `RowEllipsis`.
- **Actions & nav:** `ActionBar`, `ActionButton`, `Button`, `ButtonGroup`, `BreadCrumbs`,
  `DropdownMenu`/`DropdownMenuTrigger`, `Badge`.
- **Forms (settings, profiles, custom formats):** `Input`, `TextArea`, `Select`, `ComboBox`,
  `Checkbox`, `RadioButton`/`RadioButtonGroup`, `NumberRangeSlider`, `DatePicker`.
- **Feedback & status:** `AlertBanner`, `Message`/`MessageViewer`, `BarProgress`, `BarLoader`,
  `BlockLoader`, `MatrixLoader`, `Tooltip`, `Popover`, `Dialog`, `Drawer`, `ModalStack`/
  `ModalTrigger`, `Accordion`, `Text`, `CodeBlock`.

(Decorative/demo components — `CanvasSnake`, `Chessboard`, `ASCIICanvas`, etc. — are not used in
product screens.)

## Screen → component mapping (the v1 UI)

This is the build plan for `web/`. Each screen is assembled from the listed SRCL components only.

| Screen | Purpose | Primary SRCL components |
|--------|---------|-------------------------|
| **App shell** | nav, global actions, routing, **theme toggle (System/Light/Dark)** | `SidebarLayout`, `Navigation`, `ActionBar`, `BreadCrumbs`, `DropdownMenu`/`RadioButtonGroup` (theme) |
| **Library (Movies / TV / Music / Books)** | browse monitored items, status | `DataTable` / `SimpleTable`, `Badge`, `RowSpaceBetween`, `Input` (filter), `Select` |
| **Item detail** | one movie/series; seasons/episodes; files | `Card`/`CardDouble`, `TreeView` (seasons→episodes), `SimpleTable`, `Badge`, `ActionButton`, `Tooltip` |
| **Add / search new** | find & add a title | `Input`/`ComboBox`, `DataTable` (results), `Dialog`/`Drawer` (confirm add), `Button` |
| **Manual search / interactive grab** | pick a release, see scores | `DataTable` (releases), `Badge` (quality/score), `Popover`/`Tooltip` (why this score), `ActionButton` |
| **Activity / Queue** | in-progress downloads | `DataTable`, `BarProgress`, `BarLoader`, `RowSpaceBetween` |
| **History / Decision log** | what happened & **why** | `DataTable`, `Accordion` (expand a decision), `CodeBlock` (raw parse/score), `MessageViewer` |
| **Settings — profiles** | quality profiles | `Select`, `Checkbox`, `RadioButtonGroup`, `NumberRangeSlider`, drag-order via `ListItem`s |
| **Settings — custom formats** | CF conditions & scores | `Input`, `Select`, `ComboBox`, `Table`, `Badge`, `Dialog` |
| **Settings — indexers / clients** | integration config | `Card`, `Input`, `Select`, `Checkbox`, `AlertBanner` (test result), `Button` (test) |
| **System / status** | health, tasks, logs | `SimpleTable`, `AlertBanner`, `Message`, `BlockLoader`, `CodeBlock` |
| **First-run / onboarding** | initial setup wizard | `ModalStack`/`ModalTrigger`, `Dialog`, `Input`, `Button`, `Badge` |

The **decision-log screen is a signature feature** — it's the thing the originals can't do well.
Use `Accordion` + `CodeBlock` to let a user expand any grab/reject/upgrade and see the parsed
fields, the CF score breakdown, and the on-disk comparison that produced the verdict.

## Testing the UI

- SRCL ships with sync tests (catalog/props/theming-token guards) and a vitest suite — follow the
  same discipline for any vendored code.
- cellarr UI: component-level tests for the assembled screens (render + interaction) and a small
  set of end-to-end smoke tests driving the real API ([11-testing.md](11-testing.md)).
- **Theme tests:** assert the theme controller resolves System → the `prefers-color-scheme` class,
  reacts to a `matchMedia` change while on System, honors a persisted Light/Dark override, and sets
  `color-scheme`. Render key screens under both `theme-light` and `theme-dark` (no missing-token /
  unreadable states); a no-flash check that the initial class is set pre-hydration.
- A lint/CI check enforces the rule: **no UI primitive outside the SRCL component set.** If a PR
  introduces a non-SRCL component, CI fails. (Implementation: an allowlist of importable UI modules.)

See [`specs/web-ui.md`](specs/web-ui.md).
